# v3.6.0 — PR Review 2.0 : spec UX cible (benchmark-driven)

> Livrable n°1 du chantier v3.6.0 (roadmap.md). Synthèse de deux études menées le 2026-07-09 :
> un audit exhaustif de la surface review actuelle (baseline v2.24.0, refs `file:line` sur
> `feat/pr-review-2` @ 6c394d0) et un benchmark concurrentiel (GitKraken, Tower, Graphite,
> GitButler ; Greptile v4, CodeRabbit, Copilot code review). Cette spec fixe le périmètre et
> les décisions d'UX ; le plan d'implémentation détaillé vit dans `docs/superpowers/plans/`.

## 1. Verdict du benchmark

### Baseline GitWand (audit code)

| Axe | État actuel | Gap |
|---|---|---|
| Time-to-first-comment | Detail = `Promise.all` de 6 appels à chaque open **et** à chaque tick du poll 5 min (`usePrPanel.ts:568-575`) ; diff fetché+parsé d'un bloc, synchrone main-thread (`:631-643`, parse `:319-365`) | 3 des 6 appels sont différables par tab (`gitFileCount`, `getCIChecks`, `listIssueComments`) ; aucun résumé de PR ; l'IA est on-demand uniquement |
| Navigation grosses PRs | Rendu un-fichier-à-la-fois mais `PrInlineDiff.vue` rend **toutes** les lignes du fichier (`:336`, `:390`), zéro virtualization ; aucun viewed-state ; aucun raccourci clavier (Escape seulement) | Lag scroll sur gros fichiers ; aucun moyen de savoir où on en est dans une PR 50+ fichiers |
| Signal/bruit IA | `usePrHunkCritique.ts` : critique LLM par hunk, on-demand, verdict prose sans confiance ni ancre ligne, cache éphémère composant ; `aiFlags` de l'onglet Intelligence = heuristiques statiques (aucun LLM) ; 3 morceaux IA déconnectés | Pas de pré-review, pas de score de confiance, pas de seuil, pas de cache partagé, pas de surface settings unique |

### État de l'art 2026 (patterns retenus, sourcés)

1. **Virtualization du diff** = table stakes (GitHub l'a shippée par défaut le 2026-01-22).
2. **Keymap GitHub Files-changed** = standard de facto : `J`/`K` (hunk/fichier suivant/précédent), `V` (viewed), `Shift+V` (masquer les viewed), `T` (filtre fichiers), `?` (aide).
3. **Confiance par finding + seuil utilisateur** (Greptile : ~90 % = bug sûr, ~50-60 % = piste à vérifier).
4. **Cap délibéré du volume de commentaires** (Copilot : ~5,1 commentaires/review tenus constants ; 71 % « actionnables » comme métrique nord).
5. **Étage de vérification avant publication** (CodeRabbit : chaque suggestion validée séparément avant d'être postée) — le vrai mécanisme anti-bruit.
6. **Learn-from-rejections** (CodeRabbit Learnings) : les dismissals répétés suppriment la classe de findings.
7. **Résumé de PR en tête** (what/why/affected areas) pour réduire le time-to-understanding.
8. **Entrée par inbox triée** (Graphite « Needs your review », GitKraken Launchpad) — GitWand a déjà Today.
9. **Benchmark honnête** : réutiliser la *méthodologie* Greptile (PRs de bug-fix reproduites, « caught » = commentaire ligne explicite, catch-rate toujours apparié au noise-rate), jamais les chiffres vendeurs (82 % self-run vs 45 % en re-run indépendant).

**Positionnement GitWand** : local-first, opt-in, privé, fusionné au moteur de merge — aucun reviewer SaaS ne l'offre. La v3.6.0 vise la parité UX (perf + clavier + viewed) **plus** un pipeline IA local honnête (confiance + seuil + cap), pas la course au catch-rate.

## 2. Périmètre v3.6.0

### In scope (cette PR)

**A. Performance rebuild**
- **A1 — Parse lazy par fichier** : `getPRDiff` reste un fetch unique, mais la string unifiée est découpée par en-têtes `diff --git` (indexation légère) et chaque fichier n'est **parsé** que lorsqu'il est sélectionné (cache par fichier). Supprime le parse bloquant du PR entier à l'open. (Le vrai fetch par fichier et le Web Worker restent v3.9.0.)
- **A2 — Virtualization des lignes du diff** : `PrInlineDiff.vue` passe sur `@tanstack/vue-virtual` (déjà en dépendance) via un composable partagé `useVirtualRows` lifté du pattern hauteur-variable de `CommitLog.vue` (`useVirtualizer` + `measureElement`). Budget : DOM borné quel que soit le fichier ; fallback rendu complet sous un seuil (< 300 lignes) pour préserver find/select natif.
- **A3 — Revalidation amincie** : cœur = `getPR + listComments + listReviews` (3 appels) à l'open et au poll ; `getCIChecks` déféré au tab Checks/Diff, `gitFileCount` aux loaders lazy Intelligence, `listIssueComments` au tab Info. Écart mesurable : 6 → 3 appels sur le chemin chaud.

**B. Review flow (keyboard-first + viewed)**
- **B1 — Keymap GitHub-standard**, scopée au tab Diff, via un resolver pur testé unitairement (pattern `resolveTerminalShortcut` de `useTerminalSessions.ts:9-20`) : `J`/`K` hunk suivant/précédent (franchit les frontières de fichiers), `⇧J`/`⇧K` fichier suivant/précédent, `V` toggle viewed, `⇧V` masquer/afficher les viewed, `T` filtre fichiers, `C` commenter le hunk courant, `?` aide. Guard focus strict : inactif dans tout input/textarea/palette. Réutilise la machinerie de saut de hunk de `DiffViewer.vue:180-194`. Raccourcis ajoutés à `HelpView.vue`.
- **B2 — Viewed-file state** : persistance **client** par PR dans le cache SWR (`usePrCache`), clé `repo#number`, invalidée par fichier quand le head SHA change (re-push = fichiers modifiés redeviennent non-viewed). Checkbox par fichier dans la sidebar diff + barre de progression « X/N viewed » + option masquer les viewed. `SubmitReviewOptions` étendu d'un champ optionnel `viewedFiles?: string[]` (transport forge optionnel ; GitHub n'a pas d'API native → source de vérité = client).
- **B3 — Pending review surfacé** : badge compteur persistant (les `draftReviewComments` survivent déjà au switch de tab, le badge manque) + reprise après fermeture/réouverture du detail (draft persisté dans le cache SWR) + `⌘Enter` submit review.
- **B4 — Dismiss review + request reviewers** : nouvelles méthodes forge `dismissReview` et `requestReviewers` (GitHub réel ; GitLab réel si l'API le permet, sinon erreur honnête `ForgeNotImplementedError` — jamais de dégradation silencieuse) ; UI dans le tab Info (liste reviews + picker reviewers réutilisant `listReviewerCandidates`, déjà réel sur les 4 forges).

**C. AI pre-review pass (Greptile-style, local-first)**
- **C1 — Pipeline multi-hop local**, calculé en arrière-plan à l'ouverture d'une PR (opt-in) :
  1. *Hop diff* : hunks du PR (réutilise le parse A1).
  2. *Hop dépendances* : extraction des specifiers d'imports des fichiers touchés (`tryResolveImportConflict`/parse de `packages/core/src/resolvers/imports.ts` — extraction seule, pas de graphe : le graphe résolu est explicitement v4.0) et croisement avec les chemins du repo pour signaler « ce fichier modifié est importé par N fichiers du diff ».
  3. *Hop history* : blame des lignes modifiées via `git_blame` (`read.rs:1285`) + message du dernier commit touchant ces lignes (pattern `useBlameContext.ts`, GitHub-indépendant, marche sur les 4 forges).
  4. *Findings* : prompt structuré → JSON strict `{path, line, side, severity, confidence: 0-100, title, detail}` ; parse défensif (récup fence/brace comme `usePrHunkCritique.ts:78-99`).
- **C2 — Signal/bruit** : seuil de confiance utilisateur (défaut 60), **cap top-N par confiance** (défaut 15 findings/PR), et **mémoire locale des rejets** : un finding dismissé persiste par repo (classe = `path glob + title normalisé`) et les occurrences futures similaires sont masquées sous le seuil.
- **C3 — Exécution** : file de travail avec concurrence bornée (1 appel CLI à la fois — chaque appel = spawn subprocess), annulable (switch de PR / fermeture), gated visibilité comme le poll existant. Cache SWR par `repo#number@headSha` : jamais recalculé tant que le head ne bouge pas. Statut visible (« Pré-review : 4/12 fichiers… ») avec findings affichés au fil de l'eau.
- **C4 — Rendu** : findings ancrés ligne dans le diff via l'overlay CI-annotations généralisé (voir E1), + liste triée par confiance dans le panneau Intelligence, navigation finding-suivant/précédent au clavier (`N`/`P`).

**D. PR summary**
- **D1** : bloc « Résumé » en tête du tab Info — *quoi / pourquoi / zones affectées* — généré par le même pipeline (une passe LLM sur le diff indexé + titres de commits), caché par `headSha`, régénérable à la demande. Sert de handoff de review (copiable). Rendu via `useSafeHtml.ts` (jamais le `renderBody` hand-rolled actuel).

**E. Pipeline Intelligence unifié**
- **E1 — Type partagé `LineAnnotation`** avec `source: "ci" | "ai" | "static"`, sévérité ouverte (préserve confiance/sévérité IA au lieu de la normalisation 3-niveaux de `mapAnnotation`), matching des deux côtés old/new (le side LEFT/RIGHT des comments existe déjà comme modèle). `PrInlineDiff` consomme un flux mergé CI + IA + flags statiques, icônes/labels distincts par source.
- **E2 — Un composable `useReviewIntelligence`** qui orchestre pré-review (C), summary (D), critique de hunk existante (migrée dessus, ancrée ligne quand possible) et `aiFlags` statiques — cache partagé, état unique. `usePrPanel.ts` (1111 lignes) le **compose**, n'absorbe pas son état.
- **E3 — Une surface settings** : section unique « Review AI » dans Settings (provider hérité de `useAIProvider`, opt-in pré-review, seuil de confiance, cap, opt-in summary) — ajoutée à `AppSettings` (`useSettings.ts`) **et** `SettingsPanel.vue` (règle de sync).

**F. Multi-forge : GitLab review complet**
- **F1** : combler le plus gros stub gap — `submitReview` avec vrais commentaires inline batch (discussions API, ancres old/new correctes, plus d'objet `PrReview` synthétisé), `listReviews` réel (représenter COMMENTED/CHANGES_REQUESTED, pas approvals-only), `createComment` inline sans `as any` (ancre old-side incluse), `getFileHistory` réel.
- **F2** : Azure `updateComment`/`deleteComment` qui **throw** au runtime : soit implémentés si triviaux, soit gardés en `ForgeNotImplementedError` avec UI qui masque l'action (jamais de crash). Bitbucket/Azure inline comments = hors périmètre (roadmap : « Bitbucket/Azure follow »).

### Out of scope (explicite)

- Fetch réseau du diff par fichier + parse en Web Worker → **v3.9.0** (même hot-path, dépend de comlink).
- Graphe d'imports résolu / reverse-deps / blast-radius → **v4.0.0** (le hop dépendances de C1 reste de l'extraction+matching léger).
- Moved-code-as-moved (GitKraken), diagrammes dans le summary (CodeRabbit Atlas) → backlog, notés en veille.
- Review par commit (GitButler) / vue changeset multi-commits → **v3.12.0** (Combined Diffs).
- Complétion review Bitbucket/Azure inline → suite de F, version ultérieure.
- Harnais de benchmark automatisé type Greptile → réutilisé comme *méthodologie* de validation manuelle ici ; l'outillage viendra avec v3.7.0 (même moteur pointé sur l'index).

## 3. Métriques de succès (à vérifier avant merge)

| Métrique | Cible |
|---|---|
| Appels réseau à l'ouverture d'un detail PR (chemin chaud) | 6 → 3 |
| Parse du diff à l'open d'une PR 50+ fichiers | fichier sélectionné uniquement (plus de parse du PR entier) |
| Nœuds DOM du diff d'un fichier 5 000 lignes | borné par la virtualization (~visible + overscan), scroll fluide |
| Review complète possible au clavier seul | J/K/⇧J/⇧K/V/⇧V/T/C/N/P/⌘Enter, testés unitairement via le resolver |
| Findings IA affichés | ≤ cap (déf. 15), tous ≥ seuil de confiance, ancrés ligne, dismissables avec mémoire |
| Pipeline IA | opt-in, zéro appel si désactivé, annulé au switch de PR, jamais recalculé à headSha constant |
| Régression zéro | `pnpm -r test` vert, parité dev:web pour toute nouvelle commande, i18n 5 locales complètes |

## 4. Contraintes d'implémentation (rappel audit)

- Toute nouvelle commande Tauri → wrapper `backend*.ts` + route `dev-server.mjs` dans la même PR (parité dev:web).
- Aucun `invoke()` direct hors `backend.ts` ; aucun interval inconditionnel (tout poll IA gated visibilité + opt-in).
- Strings user-visible → clés dans les 5 locales (`en`, `fr`, `es`, `pt-BR`, `zh-CN`).
- Nouveaux settings → `AppSettings` + `SettingsPanel.vue` dans le même commit.
- HTML dérivé de contenu user/LLM → `useSafeHtml.ts` (DOMPurify).
- `packages/core` reste browser-safe (le pipeline IA vit côté desktop ; seule l'extraction d'imports est consommée depuis core).
- Fichiers pivots : `usePrPanel.ts`, `usePrHunkCritique.ts`, `useAIProvider.ts`, `usePrCache.ts`, `composables/forge/{types,GitLabProvider}.ts`, `components/{PrDetailView,PrInlineDiff,PrIntelligencePanel}.vue`, `utils/backend-pr.ts`, `dev-server.mjs`, `src-tauri/src/commands/read.rs`.
