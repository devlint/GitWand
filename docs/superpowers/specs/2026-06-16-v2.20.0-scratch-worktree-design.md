**Date:** 2026-06-16
**Branch:** `feat/v2.20.0-scratch-worktree`
**Status:** Design approved — scaffolding pass
**Roadmap source:** `ROADMAP.md` → "What's Next" → v2.20.0

---

## 1. Goal

Two related features that extend GitWand's existing merge-preview engine and
first-class worktree support (shipped v2.7):

- **A — Scratch worktree for isolated resolution.** From the merge preview or the
  Conflict Predictor, open a temporary isolated worktree
  (`gitwand-scratch-<timestamp>`) without touching the active checkout, resolve
  conflicts there, then bring the changes back to the main checkout in one click —
  with automatic cleanup on merge-back or abandonment.
- **B — Conflict Predictor extended to rebase and cherry-pick.** Generalize the
  side-effect-free predictor (`preview_merge`) to also simulate a rebase and a
  cherry-pick without mutating the working tree, with a hunk-by-hunk preview and a
  per-operation risk level (low / medium / high).

This document is the **design spec**. This branch delivers the **scaffolding**:
compiling/typechecking stubs, types, handler registration, `backend.ts` wrappers,
i18n keys in all 5 locales, and a `CHANGELOG.md [Unreleased]` entry. The full
behaviour is implemented in follow-up work guided by this spec. **No version
bump** in this pass — versioning happens at ship time via
`./scripts/bump-version.sh 2.20.0` (per `AGENTS.md`).

## 3. Feature A — Scratch worktree

### 3.1 Backend (Rust) — new module `commands/scratch.rs`

Three `#[tauri::command]` async functions. Every user-supplied path goes through
`safe_repo_path()`; all git invocations use `.args([...])` (no string
interpolation) — per `AGENTS.md` security rules.

```rust
#[derive(serde::Serialize)]
pub(crate) struct ScratchWorktree {
    pub path: String,          // absolute path of the scratch worktree
    pub branch: String,        // gitwand-scratch-<timestamp>
    pub source_branch: String, // branch/ref it was based on
    pub created_at: u64,       // unix epoch seconds
}

/// Create gitwand-scratch-<timestamp> as a sibling worktree, detached/based on
/// `source_branch` (defaults to current HEAD). Does NOT touch the active checkout.
scratch_worktree_create(cwd: String, source_branch: Option<String>) -> Result<ScratchWorktree, String>

/// Bring resolved changes from the scratch worktree back into the main checkout
/// in one operation, then remove + prune the scratch.
scratch_worktree_merge_back(cwd: String, scratch_path: String) -> Result<(), String>

/// Abandon the scratch: `worktree remove --force` + `worktree prune`.
scratch_worktree_discard(cwd: String, scratch_path: String) -> Result<(), String>
```

**Naming:** `gitwand-scratch-<unix-epoch>`. The timestamp is generated Rust-side
(`SystemTime`) so the frontend never has to. Scratch worktrees live as siblings of
the repo root (same parent dir as existing `git_worktree_add` "New task" flow).

**Merge-back strategy (spec for the impl phase, not built now):** the scratch
shares the repo's object database, so merge-back diffs the scratch tree against the
main checkout's HEAD and applies it to the working tree of `cwd` (candidate:
`git --work-tree=<cwd> checkout <scratch-branch> -- .` or an apply of the scratch
diff). Exact mechanism is decided during implementation; the stub returns
`Err("not implemented (v2.20.0)")`. Guard: refuse merge-back if the main checkout
has conflicting uncommitted changes — surface a clear error.

**Cleanup invariant:** both `merge_back` and `discard` must leave no dangling
worktree registration (always `worktree prune` after `remove`). A startup sweep of
stale `gitwand-scratch-*` worktrees is a documented follow-up (see §7).

### 3.2 Frontend

- `utils/backend.ts` — typed wrappers `scratchWorktreeCreate`,

## 4. Feature B — Extended Conflict Predictor

### 4.1 API choice (approved): separate commands

Keep `preview_merge` untouched. Add two siblings returning the **same**
`Vec<FileMergePreview>` shape so the frontend resolver path is reused verbatim:

```rust
/// Simulate rebasing HEAD onto `onto` without modifying the working tree.
preview_rebase(cwd: String, onto: String) -> Result<Vec<FileMergePreview>, String>

/// Simulate cherry-picking `commit` onto HEAD without modifying the working tree.
preview_cherry_pick(cwd: String, commit: String) -> Result<Vec<FileMergePreview>, String>
```

- `preview_rebase`: conceptually each commit `base..HEAD` replayed onto `onto`. For
  the predictor, the pragmatic approximation is a 3-way preview using
  merge-base(`HEAD`,`onto`) as ancestor (mirrors `preview_merge` with roles
  adjusted). Exact per-commit replay vs. squashed approximation is decided at impl;
  documented as a known limitation if approximated.
- `preview_cherry_pick`: 3-way merge with `commit^` as the ancestor, `HEAD` as ours,
  `commit` as theirs — reuses `merge_file_preview` directly.

Both stubs return `Err("not implemented (v2.20.0)")` in this pass.

### 4.2 Risk level

A `RiskLevel = "low" | "medium" | "high"` derived **frontend-side** in
`useMergePreview.ts` from the existing `MergePreviewSummary` (no backend change):

- **low** — no conflicting files, or all conflicting files `auto-resolved`.
- **medium** — some `partial` files, but no fully-`manual` or `add-delete` files.
- **high** — at least one `manual` or `add-delete` file.

Exposed as a computed `riskLevel` on the composable return. The hunk-by-hunk
preview reuses the existing per-file `resolve()` output (already computed) —
surfaced in the panel during impl.

### 4.3 Frontend

`useMergePreview.ts` gains an `operation: "merge" | "rebase" | "cherry-pick"`
parameter on `computePreview` (default `"merge"`, backward compatible) that
dispatches to the matching backend wrapper. `backend.ts` gets `previewRebase` and
`previewCherryPick` wrappers. The web dev mock (`dev-server.mjs`) returns an empty
preview for the two new ops (stub-safe).

## 5. Security & performance constraints (from AGENTS.md / CLAUDE.md)

- All paths via `safe_repo_path()`; all git via `.args([...])`.
- New commands are `async` (offload blocking git work) — consistent with the v2.19
  async pass.
- No new inconditional polling; no `{ deep: true }` watchers on large reactive
  structures.
- `packages/core` untouched (stays Node-free) — all new code is in `apps/desktop`.

---

## 6. Testing strategy (impl phase)

- Real temporary git repos (no git-layer mocks), per AGENTS.md.
- Scratch worktree: create → write a resolution → merge-back → assert main checkout
  updated + scratch gone + no dangling registration; create → discard → assert clean.
- Predictor: build a repo with a known rebase conflict and a known cherry-pick
  conflict; assert the preview matches the post-operation conflict set and the risk
  level is classified correctly.
- Scaffolding pass ships no new behavioural tests (stubs error out); it must not
  break `pnpm -r run build`, `pnpm test`, or `cargo check`.
