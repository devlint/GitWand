# v3.5.0 — Safety Bundle: pre-commit secrets scanner — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use `superpowers:subagent-driven-development`
> (recommended) or `superpowers:executing-plans` to implement this plan task-by-task.
> Steps use checkbox (`- [ ]`) syntax for tracking. Do the steps in order; each task ends
> with a `pnpm`/`cargo` verification command and a commit.

**Goal:** Add a zero-network, local secrets scanner to the staging area. When the user stages
changes, GitWand scans the **staged added lines** for known credential shapes (AWS, GCP, Azure,
GitHub/GitLab tokens, Slack, Stripe, OpenAI, Anthropic, RSA/OpenSSH/EC private keys, JWTs) plus
high-entropy literals. Findings surface as a **non-blocking orange badge** in the commit area;
the user can inspect them, ignore a pattern, or commit anyway (via a confirm modal). Patterns are
extensible per-repo through `.gitwandrc` (`secrets.patterns[]` + `secrets.ignore[]`). An **opt-in
pre-commit hook** (Settings > Hooks) runs the same scan from the terminal.

**Tech stack:** Rust (`apps/desktop/src-tauri`, `regex` crate), TypeScript (`packages/core`,
`packages/cli`, Vitest), Vue 3 `<script setup>` (`apps/desktop/src`), Node dev-server
(`apps/desktop/dev-server.mjs`).

## Architecture (decided during planning)

**Dual scanner, single behavioural contract.** The matching engine is implemented **twice** and
locked in sync by a parity test — the exact governance AGENTS.md already applies to the diff
parser ("both implementations must stay in sync"):

1. **Rust** (`apps/desktop/src-tauri/src/commands/secrets.rs`, `regex` crate) — the **desktop
   production path**. Satisfies the hard constraint "matching is Rust-side, zero network". The
   `regex` crate has no catastrophic backtracking (linear-time guarantee), which is a security
   plus when compiling user-supplied patterns from `.gitwandrc`.
2. **TypeScript** (`packages/core/src/secrets/`) — a browser/Node-safe mirror (pure `RegExp` +
   string math, **no Node built-ins**). Powers two things Rust can't reach:
   - `dev-server.mjs` (so `pnpm dev:web` has a **real** route, not a stub — project rule);
   - the CLI `gitwand scan` command that backs the opt-in **pre-commit hook** (runs outside the
     app, in a plain terminal `git commit`).

Both engines are **pure functions** over already-extracted input:
`scan(files: ScanFileInput[], config: SecretsScanConfig) -> SecretFinding[]`, where
`ScanFileInput = { path, addedLines: { line: number, text: string }[] }`. The **caller** does the
git extraction (`git diff --cached --unified=0`) — Rust command, dev-server route, and CLI each
run their own extraction and feed the pure scanner. This keeps the core TS scanner free of any git
dependency and makes the parity test end-to-end over a real temp repo.

**Why not a single shared pattern catalog file?** A cross-package `include_str!` from
`src-tauri` into `packages/core` is fragile (cargo doesn't track the dependency, and packaging
`src-tauri` independently breaks it), and the `regex` crate rejects constructs JS `RegExp` allows
(lookaround, backreferences). Instead: each engine keeps its own built-in table, restricted to the
**common regex subset** (character classes, anchors, quantifiers, `\b` — **no lookaround, no
backrefs**), and the **parity test over a rich fixture corpus is the contract**. This is exactly
how the diff parser dual-impl is governed today. (See Open Decisions D1/D4 for the human
checkpoint.)

**Scan target = staged added lines only.** We parse `git diff --cached --unified=0` and match
only `+` lines (never the `+++` header, never context/removed lines). Pre-existing secrets already
in history are out of scope for v3.5.0 — the feature is a *pre-commit* gate. This respects the
AGENTS.md diff gotcha (added lines start with `+`, and we must not treat `+++ ` as content).

**Secret values never leave the process boundary in the clear.** `SecretFinding` carries
`{ file, line, patternId, severity, redactedExcerpt }` — the excerpt masks the middle of the match
(first/last ≤3 chars visible). No matched value is ever `println!`/`eprintln!`/`console.log`-ed
(AGENTS.md "never log secrets").

- `packages/core` = **zero Node.js** — the TS scanner uses only `RegExp`, strings, and arithmetic.

  plain TS data.

  `crate::git::safe_repo_path` (`apps/desktop/src-tauri/src/git/cmd.rs:67`). The scan reads via
  `git diff --cached` (git confined by `.current_dir(cwd)`), not by joining paths itself.
- **No shell string interpolation** in git commands — `.args([...])` arrays only
  (`git/cmd.rs` `git_cmd()` at `:309`).
- **Secondary Rust binaries** go under `[[example]]`, never `[[bin]]` (none added here, but if a
  scan probe binary is added it follows `Cargo.toml:27`).
- **Settings** field lands in BOTH `useSettings.ts` (`AppSettings`, `:115`) and `SettingsPanel.vue`

- **i18n**: every user-visible string gets a key in all 5 locales
  (`apps/desktop/src/locales/{en,fr,es,pt-BR,zh-CN}.ts`).
- **Tests use real temp git repos** — never mock the git layer.
- **Modal confirm, not native** — reuse `askConfirm` (`App.vue:2191`, provided at `App.vue:359`).
- **Perf invariants** (apps/desktop/CLAUDE.md P6.4): the scan is triggered by staged-set changes,

## Shared type shapes (referenced across tasks)

Define once in `packages/core/src/secrets/types.ts` (Task 2) and mirror in Rust (Task 4) /
`backend.ts` (Task 6):

```typescript
export type SecretSeverity = "high" | "medium" | "low";

/** One built-in or user-supplied detector. */
export interface SecretPattern {
  id: string;              // stable id, e.g. "aws_access_key_id", "github_pat"
  regex: string;          // common Rust+JS subset — NO lookaround / backrefs
  severity: SecretSeverity;
  description: string;    // English default; UI shows i18n key by id when present
}

/** Input to the pure scanner (caller extracts this from `git diff --cached`). */
export interface ScanFileInput {
  path: string;
  addedLines: Array<{ line: number; text: string }>;
}

/** Effective config: built-ins + user extensions, resolved by the caller. */
export interface SecretsScanConfig {
  enabled: boolean;             // master switch (app setting AND .gitwandrc.secrets.enabled)
  extraPatterns: SecretPattern[]; // from .gitwandrc secrets.patterns[]
  ignore: string[];            // glob paths OR /regex/ literal-ignores, from .gitwandrc
  entropyThreshold: number;    // Shannon bits/char over long tokens; 0 disables entropy check
}

export interface SecretFinding {
  file: string;
  line: number;
  patternId: string;   // built-in id, user pattern id, or "high_entropy"
  severity: SecretSeverity;
  redactedExcerpt: string; // middle-masked; NEVER the raw secret
}
```

### Task 3 — Core TS scanner: matching + entropy + redaction + ignore

**Files:**
- Create: `packages/core/src/secrets/scanner.ts` — `scanSecrets(files, config)` +
  helpers `shannonEntropy(s)`, `redact(match)`, `isIgnored(path, value, ignore)`.
- Modify: `packages/core/src/index.ts` — re-export `scanSecrets`, catalog, and the `secrets/types`
  (so `packages/cli`, `dev-server.mjs` import from `@gitwand/core`).
- Test: `packages/core/src/__tests__/secrets/scanner.test.ts`.

**Behaviour:**
- Iterate `files → addedLines`; for each line run built-in + `config.extraPatterns` regexes; on a
  match produce a `SecretFinding` with `redactedExcerpt = redact(matchedSubstring)`.
- **Entropy pass** (only if `config.entropyThreshold > 0`): tokenize the line on
  non-`[A-Za-z0-9+/=_\-]` boundaries; for tokens length ≥ 20 whose Shannon entropy ≥ threshold,
  emit a `high_entropy` finding (severity `low`). Skip a token already covered by a regex hit on
  the same line to avoid double-reporting.
- **Ignore**: an entry that parses as `/.../` is treated as a value-regex (drop findings whose raw
  match matches it); anything else is a glob matched against `file` via the existing
  `matchGlob` from `config.ts` (import it — same package, browser-safe). Ignore is applied
  **before** redaction bookkeeping so ignored values never surface.
- **Redaction**: `redact("ghp_ABCD…WXYZ")` → `ghp…WXYZ` style (≤3 leading + `…` + ≤3 trailing;
  for very short matches, all-mask). Guarantee the raw secret is never in the returned struct.
- Cap total findings at 500 (defensive; note in a comment referencing the P6.4 payload rule).

- [ ] **Step 1 — Write failing tests** (≥ 15 cases): each built-in detected in a realistic added
  line; a user `extraPattern` detected; an `ignore` glob suppresses a path; an `ignore` `/regex/`
  suppresses a value; entropy finds a random 40-char base64 blob; entropy does **not** flag an
  English sentence or a long hex git SHA when threshold set high; redaction never contains the raw
  value; `enabled: false` → returns `[]`; findings capped at 500. Assert no finding field equals
  the planted secret verbatim.
- [ ] **Step 2 — Run, expect FAIL.**
- [ ] **Step 3 — Implement** `scanner.ts` + index re-exports.

## Open decisions for the human checkpoint (original)

- **D1 — Dual scanner (Rust + core TS mirror).** The team-lead constraint says "matching is
  Rust-side". A TS mirror in `@gitwand/core` is nonetheless required for `pnpm dev:web` (real
  route, not a stub) and for the CLI-backed pre-commit hook (runs outside the app). Recommended:
  accept the dual-impl, governed by the Task 7 parity test exactly like the existing diff-parser
  dual-impl (AGENTS.md). **Fallback if rejected:** keep matching Rust-only, implement the
  dev-server route as a JS port living *only* in `dev-server.mjs` (not core), and **defer the
  pre-commit hook to v3.5.1** (drop Tasks 12–13) since it has no non-Rust engine to call.
- **D2 — Pre-commit hook distribution.** The hook shells out to `npx @gitwand/cli scan`. Confirm
  the CLI is the intended vehicle (vs. a bundled binary). If `npx` latency is a concern, an
  alternative is documenting a globally-installed `gitwand` requirement in the install confirm.
- **D3 — Default on/off.** Proposed: scanning **on by default** (`secretsScannerEnabled: true`),
  but **never blocking** in-app (only a confirm). The pre-commit hook is separately opt-in and
  *does* block (exit 1) with a documented `--no-verify` bypass. Confirm this split.
- **D4 — No shared pattern-catalog file.** Rejected cross-package `include_str!` (fragile, and
  `regex` crate ≠ JS `RegExp` dialect); patterns are duplicated in Rust + TS and locked by parity.
  Confirm this is acceptable, or ask for a build-time codegen step to generate the Rust table from
  the TS catalog.
- **D5 — Entropy default threshold.** Proposed `4.0` bits/char over tokens ≥ 20 chars, tunable in
  Settings + `.gitwandrc`. May need tuning against false positives on real repos (base64 assets,
  minified JS); consider excluding known-generated paths via the built-in generated-file globs.
